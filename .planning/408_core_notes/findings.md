## 初始判断

- 四本均标注“2026”并带书签，优先读取 PDF 元数据、文本层和书签，不直接全量 OCR。
- 若正文已有稳定文本层，使用逐页文本提取；只对目录、公式图表或无文本页面定向 OCR。
- 成品应与数学笔记一样保留总览、章节、速查、来源说明和可折叠完整答案。
- 当前环境没有 `pdfinfo`、`pdftotext`、`mutool`、`qpdf` 或 Python 解释器，不能沿用本地 pdfplumber 路线。
- 可用工具中存在 Adobe PDF 元数据、OCR 与 Markdown 提取能力；下一步先确认其本地绝对路径参数和大文件处理能力。
- Adobe 连接器尝试上传最小的计算机网络 PDF（约 215MB）时失败，未产生远端资产；不能依赖四本大文件整体上传。
- 本机有 Node.js、npm、.NET 和 Microsoft Word COM；没有 Java、Adobe Acrobat COM 或现成的 pdfjs/pdf-parse/pdf-lib 包。
- 优先方案调整为：在任务专属工具目录安装 Mozilla PDF.js 的发布包，以流式/逐页方式提取页数、书签和文本；Word COM 只作为失败后的备选，因为全书转换开销和版式损失更大。
- 本机 Node.js 为 v24.16.0；Mozilla PDF.js 官方 releases 在 2026-07-15 显示最新稳定版为 v6.1.200，当前分支最低 Node.js 版本为 22，因此运行时兼容。
- PDF.js 官方资料：<https://github.com/mozilla/pdf.js/releases>；npm 包为 `pdfjs-dist`。
- PDF.js 6.1.200 已安装到 `.planning/408_core_notes/pdf-tools`，仅包含 3 个 npm 包，安装时禁用了 lifecycle scripts。
- 包内确认提供 `getDocument`、`getOutline`、`getTextContent`、`getPageLabels` 与目标页解析接口，可同时完成页数、书签、页码标签和逐页文本提取。
- 计算机网络 PDF 首次本地盘点成功：312 页、198 个书签项；50 个抽样页均获得文本，证明至少该书存在稳定文本层，不需要全量 OCR。
- 复核抽样数据后修正判断：50 个抽样页的 `char_count` 全为 0；计算机网络 PDF 是“图片正文 + 文本书签”，正文必须 OCR。
- 本机没有 Tesseract、ImageMagick 或 Ghostscript；PDF.js 的可选依赖已自动安装 `@napi-rs/canvas`，可在本地把指定 PDF 页渲染为 PNG，再交给 OCR。
- OCR skill 提供 SiliconFlow 的 `PaddlePaddle/PaddleOCR-VL-1.5` 接口，支持中英混排和坐标信息；默认并发上限为 5、请求上限约 60 次/分钟。
- 该 skill 的现成脚本依赖 Python，而当前环境无 Python；若检测到现有 `SILICONFLOW_API_KEY`，将按同一官方 API 协议用 Node.js 复现调用，不输出或写入密钥。
- 当前未配置 `SILICONFLOW_API_KEY`，不采用云端 PaddleOCR。
- Windows 本机自带 OCR 与 PDF WinRT 接口均可用，并已安装 `zh-Hans-CN` 简体中文识别器；首选完全本地的 Windows OCR，避免上传教材页面和产生 API 成本。
- PDF.js + Canvas 已成功把网络教材第 13 页渲染为 1353×1885 PNG（scale 2.5）；Windows OCR 最大图像边长为 10000，当前分辨率安全。
- 网络教材第 13 页本地 OCR 用时约 0.4 秒，得到 34 行、1561 字符；章节标题、考纲、定义和 TCP/IP 等主体内容可读。
- 当前 OCR 的主要噪声是中文字符间空格、水印/二维码误识别，以及 `Link/internet/Internet` 等少量英文字母混淆。成品不能直接复制 OCR 文本，必须按教材页码人工重述并校正术语。
