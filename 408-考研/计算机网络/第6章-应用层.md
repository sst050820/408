# 第6章 应用层

## 本章定位

应用层把域名、文件、邮件和 Web 需求落实为跨层交互。核心不是背端口，而是能从 DNS 开始画出 ARP、运输层连接与应用报文时间线。

> [!important] 408 必考
> DNS 递归/迭代查询、FTP 双连接、SMTP/POP3、HTTP 连接模式与 RTT 计算、常用端口、一次网页访问全过程。

- [[#网络应用模型与 DNS]] → [[#FTP 与电子邮件]] → [[#WWW 与 HTTP]] → [[#TLS、HTTP/2 与 HTTP/3]]。

## 考点地图

| 应用 | 运输层/端口 | 交互核心 |
|---|---|---|
| DNS | UDP 53 为主，TCP 53 可用 | 递归、迭代、缓存 |
| FTP | TCP 21 控制，数据连接另建 | 带外控制、主动/被动 |
| SMTP | TCP 25 | 推送邮件 |
| POP3/IMAP | TCP 110/143 | 拉取/同步邮件 |
| HTTP/1.x、HTTP/2 | TCP；明文常用 80，HTTPS 常用 443 | 请求—响应；HTTP/2 在一条 TCP 连接多路复用 |
| HTTP/3 | QUIC/UDP，HTTPS 常用 443 | HTTP 语义映射到 QUIC 流 |

## 核心知识框架

用户输入 URL 后：解析 URL → 若无缓存则 DNS 得 IP → 判断同网并可能 ARP 得下一跳 MAC → 建运输层连接 → 若 HTTPS 则协商 TLS → 发送 HTTP 请求 → 收响应并解析对象 → 对新增对象继续请求。题目给定缓存、连接复用、并行度和 RTT 才能确定总时间。

## 完整知识点

### 网络应用模型与 DNS

C/S 中服务器常开并被动接收请求，客户主动联系；P2P 节点兼具客户与服务器角色，可扩展性较好但管理复杂。

DNS 是分布式、层次化命名系统。域名自右向左层次升高，根之下是顶级域、二级域和各级子域；完整域名可用末尾点表示根。

服务器层次：根、顶级域、权限域名服务器；本地域名服务器通常代表主机执行查询，但不属于域名层次树的权威层。资源记录常见：A（IPv4）、AAAA（IPv6）、NS（权威服务器）、CNAME（别名）、MX（邮件服务器）。

- 递归查询：受询服务器代查询者继续查并返回最终结果。
- 迭代查询：受询服务器返回下一步应询问的服务器。
- 主机到本地 DNS 通常递归，本地 DNS 向根、TLD、权威服务器通常迭代。
- 缓存按 TTL 失效。DNS 常用 UDP 53；响应过大、区域传送或需要可靠连接时可用 TCP 53。

DNS 时延计算必须按实际查询链逐段求和。令主机到本地 DNS 的一次请求—响应往返为 $RTT_L$，本地 DNS 到根、顶级域、权限服务器的往返依次为 $RTT_R$、$RTT_T$、$RTT_A$。在串行迭代、各级均未命中缓存、忽略处理与报文发送时延时：

$$
T_{DNS}=RTT_L+RTT_R+RTT_T+RTT_A
$$

各服务器距离与路径不同，不能默认这些 RTT 相等；题目若给出多级子域或额外转介，就继续累加对应 RTT。若本地 DNS 缓存命中有效记录，只需 $RTT_L$；若浏览器、操作系统或本机缓存直接命中，则不产生网络 DNS 查询，网络 DNS 时延为 0。若题目给出并行查询、重传或 TCP 建连，应按时间线另行加入，不能套用串行求和式。

> [!info] 技术更新：现代 DNS
> DoT/DoH 为 DNS 增加加密运输，DNSSEC 提供数据来源认证与完整性验证；它们改变安全与承载，不改变 408 的层次域名、递归/迭代与缓存主线。

### FTP 与电子邮件

FTP 使用两条 TCP 连接：控制连接在整个会话保持，服务器熟知端口 21；每次文件传输另建数据连接。主动模式由服务器端口 20 主动连客户指定端口；被动模式由服务器告知临时端口，客户主动连接，更适合客户位于 NAT/防火墙之后。控制信息与数据分离称带外控制。

电子邮件系统由用户代理、邮件服务器和协议组成。教材常把一封电子邮件总括为“信封和内容”，内容再分首部与正文；协议精确分析时必须区分 SMTP 传输信封与邮件内容。MIME 扩展内容类型与编码，使非 ASCII 文本和附件可传送。

> [!info] 技术更新/格式核验：传输信封与邮件内容
> [RFC 5321](https://www.rfc-editor.org/rfc/rfc5321) 的 SMTP 传输信封由会话中的 `MAIL FROM` 与一个或多个 `RCPT TO` 等命令给出，供服务器路由和投递，不等同于邮件正文中的 `From:`、`To:` 字段。[RFC 5322](https://www.rfc-editor.org/rfc/rfc5322) 定义 Internet Message Format，即首部字段、空行和正文。408 若问教材组成，按“信封+内容（首部+正文）”作答；若给 SMTP 命令或要求分析格式，就明确分开传输信封与 RFC 5322 首部/正文。

发送链：发件人用户代理 → 发件服务器 → 收件服务器，通常用 SMTP 推送；收件人用 POP3 拉取或 IMAP 同步管理。SMTP 基于 TCP、命令/响应交互；它不负责最终阅读。POP3 偏下载，IMAP 保留服务器文件夹与状态。浏览器邮件在用户与 Web 邮件服务器间使用 HTTP，但服务器之间仍可使用 SMTP。

### WWW 与 HTTP

URL 由方案、主机、可选端口、路径、查询等组成。HTTP 是无状态的请求—响应协议，Cookie/会话机制可在应用层维护状态。

HTTP 请求报文由请求行、首部字段、空行、可选内容组成；响应由状态行、首部、空行、可选内容组成。常用方法 GET、HEAD、POST、PUT、DELETE；状态码类别为 1xx 信息、2xx 成功、3xx 重定向、4xx 客户端错误、5xx 服务器错误。

> [!info] 技术更新：HTTP 语义
> [RFC 9110](https://www.rfc-editor.org/rfc/rfc9110) 定义 HTTP 的核心语义，包括方法、状态码、字段以及请求和响应的解释；HTTP 缓存规范明确见 RFC 9111。408 仍以经典请求/响应格式和连接时延为主。

HTTP/1.0 常按非持续连接分析：每个对象单独 TCP 连接；无并行、忽略传输时延时，一个对象通常需 1 RTT 建 TCP + 1 RTT 请求/首字节响应。HTTP/1.1 默认持续连接，可在同一连接发送多个请求；题目若规定流水线，可连续发请求，否则逐个等待响应。

基础对象引用 $n$ 个小对象，在已知服务器、忽略 DNS/传输时延时：

- 非持续、串行：$2(n+1)RTT$。
- 非持续、$n$ 个对象完全并行：基础对象 $2RTT$，并行阶段 $2RTT$，共 $4RTT$。
- 持续、无流水线：建连 $1RTT$，基础对象 $1RTT$，再 $nRTT$，共 $(n+2)RTT$。
- 持续、流水线：建连 $1RTT$，基础对象 $1RTT$，成批请求 $1RTT$，共 $3RTT$。

> [!warning] 公式条件
> 上述只适用于小对象、同一服务器、忽略数据发送时间和处理时间。若 DNS、TLS、不同服务器、连接数上限或对象较大，必须逐项加入。

缓存通过新鲜度与条件请求减少流量；条件 GET 可携带验证信息，未修改时返回 304。代理缓存位于客户和源服务器之间，既可缩短时延，也可降低接入链路负载。

> [!info] 技术更新：HTTP 缓存语义
> [RFC 9111](https://www.rfc-editor.org/rfc/rfc9111) 专门规定 HTTP 缓存、年龄、新鲜度与验证。它用于核验现代缓存语义；408 重点仍是代理缓存效果、条件请求与 304 响应。

### TLS、HTTP/2 与 HTTP/3

> [!info] 技术更新：TLS 1.3
> [RFC 8446](https://www.rfc-editor.org/rfc/rfc8446) 定义 TLS 1.3。TLS 位于应用与运输服务之间，为 HTTP 提供机密性、完整性和端点认证。典型 Web 握手由服务器出示证书并完成服务器认证；客户端证书认证是可选项，只有服务器请求且部署要求时才进行。408 经典 HTTP 时延题只有明确给出 TLS 时才加入握手时间。

> [!info] 技术更新：HTTP/2
> [RFC 9113](https://www.rfc-editor.org/rfc/rfc9113) 在一个连接上用二进制帧、多路复用与首部压缩承载多个流，减少 HTTP/1.1 应用层队头阻塞；TCP 丢包仍可能阻塞同连接各流。

> [!info] 技术更新：QUIC 与 HTTP/3
> [RFC 9000](https://www.rfc-editor.org/rfc/rfc9000) 定义基于 UDP 的 QUIC 运输；[RFC 9114](https://www.rfc-editor.org/rfc/rfc9114) 定义 HTTP/3。QUIC 集成 TLS 1.3、可靠流与拥塞控制；某流的 STREAM 数据丢失不会阻塞其他流按各自顺序交付，丢失数据由发送端装入新 QUIC 包再次发送。它不表示 UDP 本身变可靠，而是 QUIC 在 UDP 之上实现可靠性。

### 综合交互时间线

1. 主机无 IP 配置时先 DHCP；若已配置则跳过。
2. DNS 缓存未命中：主机询问本地 DNS，本地 DNS 迭代查询根、TLD、权威服务器。
3. 主机判断 Web 服务器是否同网段；若异网段，ARP 查询默认网关 MAC。
4. Ethernet 帧送达网关；每跳路由器更换二层首部、TTL 减 1，IP 目的地址保持服务器地址。
5. TCP 三次握手；HTTPS 再进行 TLS 握手。HTTP/3 则建立 QUIC 连接。
6. 客户发送 HTTP 请求，服务器返回基础 HTML；浏览器解析并请求嵌入对象，按连接复用与并行条件计时。

## 典型题型与方法

1. **端口题**：先确定协议与运输层，再区分服务器熟知端口和客户临时端口。
2. **DNS 题**：画主机、本地、根、TLD、权威五列，标递归/迭代与缓存命中。
3. **FTP 题**：控制连接贯穿会话，数据连接按传输建立；主动/被动看谁发起数据连接。
4. **HTTP RTT**：列 DNS、TCP、TLS、请求响应四段，再按持续、并行、流水条件合并。
5. **综合题**：严格按 DHCP→DNS→ARP→路由→TCP/TLS→HTTP 顺序，并标每层地址是否改变。

## 易错点

- DNS 既可能用 UDP 也可能用 TCP；不能只凭端口 53 判断运输层方式。
- 本地 DNS 不一定是权威服务器；递归与迭代描述的是查询责任，不是服务器级别。
- FTP 控制连接与数据连接端口、持续时间不同。
- SMTP 用于发送/服务器转发，POP3/IMAP 用于接收或同步。
- HTTP 无状态不等于 TCP 无连接；Cookie 可保存会话状态。
- HTTP/2 多路复用不等于没有 TCP 层队头阻塞；HTTP/3 的可靠性由 QUIC 提供。

## 跨章节/跨科联系

- 与第 3、4 章：DNS 得 IP 后仍需 ARP 与逐跳转发。
- 与第 5 章：HTTP/1.x、HTTP/2、SMTP、FTP 通常使用 TCP；DNS 常用 UDP；HTTP/3 使用 QUIC/UDP。
- 与信息安全：TLS 认证、密钥协商与加密保护 HTTP 内容。
