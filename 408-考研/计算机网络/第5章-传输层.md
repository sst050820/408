# 第5章 传输层

## 本章定位

传输层把主机到主机扩展为进程到进程通信。TCP 的连接、字节编号、累计确认、流量控制、拥塞控制和 RTT 是综合计算主线。

> [!important] 408 必考
> UDP/TCP 首部、三次握手与四次挥手、序号/ACK、超时与快速重传、接收窗口、拥塞窗口及状态变化。

## 考点地图

| 对象 | 数值单位 | 谁产生/限制 |
|---|---|---|
| 序号 seq | 字节 | 发送方：本段首数据字节序号 |
| 确认号 ack | 字节 | 接收方：下一个期望字节序号 |
| rwnd | 字节 | 接收方缓存可用空间 |
| cwnd | 字节或 MSS | 发送方按网络拥塞状态维护 |
| RTO | 时间 | 由采样 RTT 平滑估计 |

## 核心知识框架

应用进程通过端口被标识。UDP 保留报文边界并尽力交付；TCP 建立全双工字节流连接，以序号、确认、校验、计时器和重传实现可靠性，以 rwnd 保护接收方，以 cwnd 和拥塞算法保护网络。

## 完整知识点

### 传输层服务与 UDP

- 套接字端点可表示为 `(IP 地址, 端口号)`；TCP 连接由源/目的 IP 与源/目的端口四元组标识。
- 端口 0～1023 为熟知端口，服务器通常使用固定端口，客户临时选用动态端口。复用是多进程交给传输层，分用是按端口交给正确进程。
- UDP 无连接、尽最大努力、面向报文、首部 8 B，支持一对一、一对多、多对一和多对多；应用可自建可靠机制。
- UDP 首部：源端口、目的端口、长度、检验和，各 2 B；长度含首部与数据。
- UDP 检验和覆盖伪首部、UDP 首部与数据；伪首部只参与计算，不传输。按 16 bit 反码求和再取反；数据为奇数字节时补零仅用于计算。在 IPv4 中 UDP 检验和可选，字段全 0 表示发送方未计算；在 IPv6 中 UDP 检验和通常必须使用，不能以全 0 表示省去校验。

> [!info] 技术更新：基于 UDP 的现代运输
> [RFC 9000](https://www.rfc-editor.org/rfc/rfc9000) 定义的 QUIC 在 UDP 之上实现连接建立、可靠流、拥塞控制并集成 TLS。某一流的 STREAM 数据丢失不会阻塞其他流按各自顺序交付；丢失的 STREAM 数据可由发送端装入新的 QUIC 包再次发送。它属于理解现实网络的技术更新，不改变 408 对 UDP 自身“无连接、尽最大努力”的经典口径。

### TCP 报文与连接管理

TCP 面向连接、可靠、全双工、面向字节流、点对点。基本首部 20 B，字段含源/目的端口、序号、确认号、数据偏移、控制位、窗口、检验和、紧急指针和选项。MSS 指 TCP 数据最大长度，不含 TCP/IP 首部。

| TCP 字段 | 位宽 | 单位、作用与 408 边界 |
|---|---:|---|
| 源/目的端口 | 各 16 bit | 标识通信进程 |
| 序号 | 32 bit | 本段数据第一个字节的序号，按模 $2^{32}$ 循环 |
| 确认号 | 32 bit | ACK=1 时有效，表示下一个期望字节序号 |
| 数据偏移 | 4 bit | 以 4 B 字为单位；最小 5 即 20 B，最大 15 即 60 B |
| 保留与控制位 | 经典题常按 6 bit 控制位 | 408 核心为 URG、ACK、PSH、RST、SYN、FIN；新增标志属于扩展边界，按题给格式处理 |
| 窗口 | 16 bit | 单位 B，首部原字段最大 65535；窗口扩大选项可放大有效 rwnd，但不改变字段位宽 |
| 检验和 | 16 bit | 覆盖 TCP 伪首部、TCP 首部和数据，必须使用 |
| 紧急指针 | 16 bit | URG=1 时解释其边界，经典题按教材定义 |
| 选项与填充 | 0～40 B | 使首部按 4 B 对齐；MSS、窗口扩大、时间戳、SACK 等按题设使用 |

> [!warning] 408 经典与扩展边界
> 无选项时 TCP 首部取 20 B，经典控制位取 6 个，窗口题先按 16 bit 通告值计算。只有题目明确给出窗口扩大、SACK、时间戳或新增控制位时才引入扩展，不自行改变教材报文结构。

关键控制位：SYN 建立连接，FIN 释放发送方向，ACK 表示确认号有效，RST 复位，PSH 提示尽快交付，URG 配合紧急指针。

三次握手时间线：

1. 客户 `SYN=1, seq=x`，进入 SYN-SENT；SYN 消耗一个序号。
2. 服务器 `SYN=1, ACK=1, seq=y, ack=x+1`，进入 SYN-RCVD。
3. 客户 `ACK=1, seq=x+1, ack=y+1`，双方进入 ESTABLISHED；纯 ACK 不消耗序号。

三次而非两次可使双方确认收发能力，并避免失效旧连接请求造成错误建立。

四次挥手：主动方 FIN；被动方 ACK；被动方数据发送完再 FIN；主动方 ACK 后进入 TIME-WAIT，等待 $2MSL$。FIN 即使不携带数据也消耗一个序号。TIME-WAIT 既保证最终 ACK 丢失时可重发，也让旧报文在网络中消失。TCP 可半关闭，因此通常需要四段。

### 可靠传输与 RTT

TCP 对字节编号。若一段 `seq=S` 携带 $L$ B 数据，则下一新数据段序号为 $S+L$；SYN、FIN 各额外占一个序号。`ack=A` 表示 $A$ 之前所有字节均已按序收到，下一期待 $A$，是累计确认。

接收失序段的处理实现可异，但 408 常按累计确认：重复发送当前期望字节的 ACK；收到缺口后确认号一次前移。发送方对最早未确认数据维护重传计时器；超时重传，收到三个重复 ACK 可快速重传缺失段。

RTT 估计：

$$
\text{EstimatedRTT}=(1-\alpha)\text{EstimatedRTT}+\alpha\text{SampleRTT}
$$

$$
\text{DevRTT}=(1-\beta)\text{DevRTT}+\beta|\text{SampleRTT}-\text{EstimatedRTT}|
$$

$$
RTO=\text{EstimatedRTT}+4\text{DevRTT}
$$

重传段的 ACK 无法判定对应原段还是重传段，Karn 思想是不用该样本更新 RTT，并在超时后增大 RTO。

### 流量控制与拥塞控制

接收方在窗口字段通告 rwnd。发送方未确认数据不得超过有效发送窗口：

$$
W_{send}=\min(rwnd,cwnd)
$$

零窗口时发送方用持续计时器定期发送窗口探测，防止窗口更新报文丢失造成死锁。糊涂窗口问题可由接收方等待较大空间、发送方尽量聚合小数据来缓解。

拥塞控制关注网络，流量控制关注接收端。408 经典算法：

- **慢开始**：每收到覆盖新数据的 ACK，cwnd 增长；按 RTT 观察近似翻倍，直到 ssthresh。
- **拥塞避免**：每 RTT 近似增加 1 MSS，即加法增大。
- **超时**：`ssthresh=max(FlightSize/2,2MSS)`，cwnd 置 1 MSS，重新慢开始。
- **三个重复 ACK（Reno）**：先令 `ssthresh=max(FlightSize/2,2MSS)`，立即快重传缺失段，再令 `cwnd=ssthresh+3MSS` 进入快恢复；加上的 3 MSS 对应已离开网络且触发重复 ACK 的三个报文段。快恢复期间每收到一个**额外重复 ACK**，令 `cwnd=cwnd+MSS`，窗口许可时可再发送一个新段。收到能确认重传段的**新 ACK** 后，令 `cwnd=ssthresh`，退出快恢复并进入拥塞避免。

> [!warning] 口径
> 题目可能按“轮次”给 cwnd，以 MSS 为单位。必须先确认事件发生在该轮发送前还是结束后、采用 Reno 还是 Tahoe、门限取旧 cwnd 一半还是在途量一半。若题目或指定教材采用“收到三个重复 ACK 后直接令 cwnd=ssthresh，并进入拥塞避免”的简化模型，优先按题设/教材计算；若明确要求完整 Reno，才采用上面的膨胀窗口与快恢复步骤。Tahoe 在丢包后将 cwnd 置 1 MSS。

公平性方面，多条长期 TCP 连接通过加法增大、乘法减小趋向共享瓶颈；UDP 本身不执行 TCP 拥塞控制。

> [!info] 理解补充
> TCP 的累计 ACK 与 GBN 外观相近，但 TCP 是字节流，可选择性确认并可能缓存失序数据；不能机械地把 TCP 等同为 GBN。

## 典型题型与方法

1. **序号题**：画每段 `[首字节, 末字节]`，ACK 写“下一个期望字节”；单列 SYN/FIN 的 1 个序号。
2. **连接题**：逐段写标志、seq、ack 和状态，纯 ACK 不增加序号。
3. **窗口题**：计算已发未确认字节、rwnd 与 cwnd，取最小值后减在途数据。
4. **拥塞题**：列轮次表，慢开始指数、拥塞避免线性；在丢失事件点按题设更新 cwnd/ssthresh。
5. **RTO 题**：按给定 $\alpha,\beta$ 的更新顺序计算，避免把 SampleRTT 直接当 RTO。

## 易错点

- TCP 序号给字节，不给报文段；ACK 确认的是下一个期望字节。
- 接收窗口字段不等于发送方当前可再发送量，还要扣除未确认在途数据。
- MSS 不含 TCP/IP 首部；MTU 是链路层净荷对 IP 数据报的限制。
- 超时与三个重复 ACK 对 cwnd 的处理不同。
- UDP 检验和含伪首部，但伪首部不在线上传送。
- TIME-WAIT 出现在主动关闭方。
